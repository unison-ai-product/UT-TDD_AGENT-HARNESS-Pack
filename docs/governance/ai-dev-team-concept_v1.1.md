# 目次


# エグゼクティブサマリ

本構想書は、AI実装エージェントを開発の主体とし、人間は上流(技術仕様)と下流(品質・運用)の判断に集中するシステム開発チームの構成を定義する。GitHubを中心とした統合基盤、ハーネスエンジニアリングによる品質保証、TDD駆動開発を前提とした自動化フローを採用することで、少人数で複数プロダクトを並行運営できる開発体制を実現する。

## チームの3層構造

コア層: 技術責任者(TL)、品質責任者(QA)、UI/UXデザイン — 戦略・方針を定めるコア

変動層: AI実装・保守(テスト指示・運用監視) — 並列度に応じて増減

自動層: AI実装エージェント + CI/CD + ハーネス — 実装と検証の自動化

## 責任の二極化

人間の責任を上流(TL: 技術正しさ)と下流(QA: 品質正しさ)に分離する。実装はAIに委譲し、CIで自動検証する。判断と実装と品質保証が独立して並行進化する。

## セキュリティ統合

セキュリティは独立した役割ではなく、開発フローの全工程(開発時/コミット時/CI時/デプロイ時/運用時)に組み込む。Shift-Leftの思想により、AI生成コードの脆弱性を早期に検出する。


# 1. 基本理念

## 1.1 設計思想

本チーム構成は以下の5つの原則に基づいて設計されている。

人間とAIの役割分担: 判断は人間、実装はAI

責任の二極化: 上流(TL)と下流(QA)を対称に配置

固定と変動の分離: コアは少人数、AI実装・保守は可変

単一の真実の源: GitHubに全情報を集約

セキュリティ・バイ・デザイン: 全工程に統合

## 1.2 採用する業界手法

| 手法 | 概要 | 本チームでの位置づけ |
| --- | --- | --- |
| TDD(テスト駆動開発) | テスト先行で実装を生成 | AI実装の品質ゲート |
| Spec-Driven Development | 仕様 → テスト → 実装の順 | TLの中核業務 |
| GitHub Flow | main + 短命ブランチの軽量フロー | 標準ブランチ戦略 |
| Harness Engineering | AIエージェントを安全に動かす土台 | TLの責任領域 |
| GitOps | Gitを真実の源として運用 | 運用原則 |

## 1.3 役割設計の根本原則

責任範囲は一意かつ明確

人間が担う領域は AI で代替できない判断業務

実装作業はすべて AI への指示として実行する

各役割の境界はドキュメント(AGENTS.md / SKILL.md / ADR)で明文化する

| 💡 チーム設計の本質<br>コードを書く人を雇うのではなく、AIが書いたコードの妥当性を判断する人を配置する。<br>人数を増やしてもコミュニケーションコストが指数関数的に増えない構造を目指す。 |

# 3. セキュリティ統合

## 3.1 基本思想: Shift-Left Security

セキュリティを開発プロセスの後工程ではなく、全工程に統合する。AI生成コードのリスクを前提に、各レイヤーで多層的な防御を構築する。

| ⚠️ AI時代の主要セキュリティリスク<br>AI生成コードに含まれる既知脆弱性パターン<br>シークレット情報の意図しない露出(コード・ログ・プロンプト)<br>プロンプトインジェクションによるAI誤動作<br>AIによる不適切な依存ライブラリ採用<br>コード生成の網羅性の見落としによる権限・認可漏れ |
| --- |

## 3.2 セキュリティの責任配分

| 役割 | セキュリティ責任 |
| --- | --- |
| TL | アーキテクチャレベルのセキュリティ設計、ハーネスへのセキュリティ統合、セキュリティ規約のAGENTS.md記述 |
| QA | セキュリティテスト整備、運用監視、脆弱性対応、インシデント対応プロセス整備 |
| AI実装・保守 | 日常的なセキュリティアラートの初期対応、エスカレーション |
| AI実装層 | 事前定義された規約に従う(自律判断はしない) |

## 3.3 多層防御の設計

### 3.3.1 開発時(Develop)

AGENTS.md / CLAUDE.md にセキュリティ規約を明記

禁止パターン: 生SQL、文字列展開によるコマンド実行、機密情報のログ出力

必須パターン: パラメータ化クエリ、入力検証、認可チェック

IDE内セキュリティチェッカー(Snyk、SonarLint等)

### 3.3.2 コミット時(Commit)

pre-commit フックでシークレット検出(gitleaks、detect-secrets)

コミット署名(GPG/SSH署名)の必須化

.gitignore の徹底(.env、credentials、key ファイル等)

### 3.3.3 CI時(Build)

SAST: 静的セキュリティ解析(CodeQL、Semgrep)

SCA: 依存ライブラリの脆弱性スキャン(Dependabot、Snyk、Trivy)

シークレットスキャン(GitHub Secret Scanning)

コンテナイメージスキャン(Trivy、Grype)

ライセンスチェック(license-checker)

### 3.3.4 デプロイ時(Deploy)

インフラのIaCセキュリティチェック(tfsec、Checkov)

シークレット管理(GitHub Secrets、AWS Secrets Manager、1Password)

最小権限の原則(IAMロール設計)

本番昇格は人間判断を残す

### 3.3.5 運用時(Operate)

DAST: 動的セキュリティ解析(OWASP ZAP等)

WAF / IDS / IPS の設定

監査ログ収集と異常検知

インシデント対応プロセス(IRP)の整備と定期訓練

## 3.4 AI 特有のセキュリティ対策

### 3.4.1 プロンプト・指示の保護

社外秘・個人情報・顧客データをAIへ送信する前のフィルタリング

Zero Data Retention(ZDR)契約のあるAIサービス利用

ローカルLLMの活用(機密度が高い処理)

プロンプトインジェクション対策(入力サニタイズ)

### 3.4.2 AI生成コードの検証

AI生成コードであることの透明化(PR説明欄に明記)

AIレビュー + 人間レビューの2段階チェック

セキュリティ観点のテストを必ず含める(SQLi、XSS、CSRF、認可チェック)

### 3.4.3 サプライチェーンセキュリティ

AIが提案した新規依存ライブラリは必ずレビュー

SBOM(Software Bill of Materials)の自動生成

依存元の信頼性確認(GitHub星数、メンテナンス状況、ライセンス)

## 3.5 アクセス権限の管理

最小権限の原則(Principle of Least Privilege)

ロールベースアクセス制御(RBAC)

MFA(多要素認証)の必須化

パスワードマネージャ(1Password Teams等)で認証情報一元管理

退場時の即日アクセス剥奪手順

AIエージェント用アカウントの分離と権限制限

## 3.6 インシデント対応プロセス

検知: 監視システムまたは外部報告によるインシデント検知

初動: AI実装・保守 → QA → TLへのエスカレーション

影響範囲特定: 何が漏れた/壊れたかをログから特定

封じ込め: 該当機能の停止、アクセス遮断、認証情報のローテーション

根本原因分析: 5 Whys 等で原因を特定

修正と検証: 修正PR作成、リグレッションテスト追加

再発防止: AGENTS.md、CIワークフロー、運用手順への反映

報告: 関係者への正確な事実報告


# 4. 採用ツール一覧

業界データでは、10名未満のスタートアップが日常的に使うSaaSツールは10種類前後が標準。100以上のツールを契約しているケースが多いが、その大半は使われていない。本章では「実際に毎日動かす最小セット」と「必要が見えてから足すもの」に分けて整理する。

| 💡 実態と原則<br>業界調査: 早期スタートアップ(10名未満)が日常使うSaaSは10種類前後。<br>本構想の最小セットも同水準(10種類前後)に収める。<br>「カテゴリごとに1つ」を原則とする。AI実装ツールだけは例外的に2つ併用が標準。<br>拡張ツールは必要が見えてから入れる。最初から導入しない。 |

## 4.4 ツール選定の3原則

原則1: カテゴリごとに1ツール。AI実装だけ例外として2つ併用。

原則2: GitHubで完結できる機能は GitHub 内で完結させる(Issue管理、CI/CD、ドキュメント等)。

原則3: 新規ツール導入時は、既存ツールで代替できないかを必ず確認する。

### 4.4.1 国産か海外かの判断軸

| 観点 | 国産ツールを選ぶ | 海外ツールを選ぶ |
| --- | --- | --- |
| 顧客要件 | 国内データ保管が必須 | グローバル展開を視野 |
| 業界 | 金融・医療・公共 | IT・スタートアップ全般 |
| 商習慣 | 請求書・印鑑・源泉徴収対応 | シンプルな取引が中心 |
| 機能性 | (機能差を許容) | 最新機能を重視 |
| サポート | 日本語必須 | 英語可 |

ソースコードの学習利用オプトアウト設定が可能か

監査ログの取得範囲

利用規約に商用利用制限がないか

## 6.2 構成パターン詳細

### 6.2.1 Solo + AI

1名(全責任兼任)+ 技術メンター(副業、月数時間)

AI(Claude Code、Codex)が実装担当

到達目標: GitHub中心フロー確立、AGENTS.md整備、最小CI運用

### 6.2.2 最小チーム

TL: 1名

QA: 1名(AI実装・保守業務を兼任)

UI/UXデザイン: 業務委託

到達目標: PR運用本格化、ブランチ保護、カバレッジゲート、セキュリティスキャンCI統合

追加導入ツール: 1Password Teams、CodeQL/Snyk、Dependabot、Sentry

### 6.2.3 専任化

TL: 1名

QA: 1名

AI実装・保守: 1-2名

UI/UXデザイナー: 1名

到達目標: 各役割の専任化、ハーネス本格運用、ドキュメント整備の定常化

追加導入ツール: Notion(ナレッジ管理)、Cloudflare WAF、Datadog/Grafana

### 6.2.4 拡張

TLチーム: 2-3名(うちハーネス改善担当を含む)

QAチーム: 2-3名(うちドキュメント・ナレッジ担当を含む)

AI実装・保守: 2-3名(複数AI並列運用)

UI/UXデザイン: 1-2名

到達目標: ハーネス改善の継続運用、ナレッジ蓄積体制、複数プロダクト並行運用

追加導入ツール: PagerDuty、SBOM管理、より高度な監査ログ

## 6.3 拡張のトリガー

構成の見直しは、人数ではなく以下の事象で判断する。

| 拡張トリガー | 判断目安 |
| --- | --- |
| Solo → 最小チームへ | 1人で意思決定速度が落ちる/技術判断に困る場面が増える |
| 最小 → 専任化へ | プロダクト数が3つ以上 / TL/QAが兼任しきれない |
| 専任化 → 拡張へ | 並列AI数が増えAI実装・保守1名で捌けない / プロダクト数が5つ以上 |


# 7. リスクと対策

## 7.1 主要リスク

| リスク | 影響 | 対策 |
| --- | --- | --- |
| AI出力の品質劣化 | 高 | ハーネスの強化、AI実装・保守の観点リスト整備、AI複数並列で結果比較 |
| Bus Factor=1(属人化) | 高 | ドキュメント整備、ペアプロ、ナレッジ移転、AGENTS.md整備 |
| セキュリティ事故 | 致命的 | 多層防御、定期監査、インシデント訓練 |
| AIコスト爆発 | 中 | API使用量モニタリング、予算アラート、モデル選定最適化 |
| AI生成コードの法的問題 | 中 | ライセンスチェック自動化、AI生成範囲の透明化 |
| 人材採用の難航 | 中 | 副業・業務委託の活用、AI実装・保守の育成プログラム |
| プロセス硬直化 | 中 | 定期レトロスペクティブ、ハーネスの継続改善 |

## 7.2 セキュリティリスクの追加対策

年1回以上のセキュリティ監査(外部監査含む)

ペネトレーションテストの定期実施

セキュリティインシデント保険の加入

GDPR / 個人情報保護法 / その他規制の遵守チェック

## 7.3 事業継続(BCP)の観点

バックアップ: 全データの自動バックアップ、定期リストアテスト

災害対策: マルチリージョン展開、オフサイトバックアップ

キーマン不在対応: 役割の冗長化、ナレッジの形式知化

| ⚠️ リスク管理の原則<br>リスクをゼロにすることは不可能。受容可能なレベルまで下げ、残余リスクは明示的に受容する。<br>対策コストとリスク影響のバランスを見極める。<br>リスク評価は定期的に見直す。 |

# 9. 立ち上げアクションプラン

人を入れる前の基盤整備(フェーズ0)から開始する。基盤が整ってから採用に進む順序を守ることで、入った人がすぐに動ける状態を作る。

## 9.1 フェーズ0: 基盤整備(人を入れる前にやる)

採用や本格的な開発に進む前の準備作業。1人または既存メンバーで完了させる。

### 9.1.1 GitHub 組織の整備

GitHub 組織アカウントを作成する(個人アカウントから組織へ)

リポジトリ命名規則を決める(例: <product>-<component>)

既存リポジトリを組織アカウントに移管する

メンバー権限ロールを定義する(Owner / Maintainer / Member)

### 9.1.2 リポジトリ標準ファイルの整備

全リポジトリに以下を配置する。テンプレートリポジトリを1つ作っておくと展開が早い。

AGENTS.md(AIエージェントへの指示)

CLAUDE.md(Claude Code 専用指示)

README.md(プロジェクト概要)

SECURITY.md(脆弱性報告窓口)

.gitignore(言語別)

pull request template: `.github/pull_request_template.md`

.github/dependabot.yml

.github/workflows/ci.yml(最小CI)

### 9.1.3 ブランチ運用ルールの設定

main ブランチに保護設定を入れる(直接push禁止、PR必須、CI通過必須)

ブランチ命名規則を AGENTS.md に明記する

Conventional Commits 規約を AGENTS.md に明記する

マージ方式を統一する(Squash and Merge 推奨)

### 9.1.4 シークレット管理の整備

1Password 等のパスワードマネージャを導入する

既存の認証情報をすべて移行する

GitHub Secrets を使った CI への安全な渡し方を整備する

.env ファイルが .gitignore に含まれていることを確認する

### 9.1.5 最小CIの稼働

Lint / Format チェックを CI に組み込む

ユニットテスト実行を CI に組み込む

Dependabot を有効化する

GitHub Secret Scanning を有効化する(設定で有効化するだけ)

## 9.2 フェーズ1: 立ち上げ

基盤が整ったら、最初の構成パターン(Solo + AI)で実運用を開始する。

AI実装ツール(Claude Code および Codex)を導入

Figma 等のデザインツールを導入(必要なら)

最初のプロダクト開発を AGENTS.md ベースで進める

PR 運用、CI フィードバックを実体験する

運用上の課題を AGENTS.md にフィードバックする

## 9.3 フェーズ2: 品質ゲートの強化

運用が安定してきたら、品質保証の仕組みを段階的に強化する。

テストカバレッジ計測の導入(Codecov 等)

カバレッジゲートを CI に設定(80%以上など)

AI レビューの有効化(Claude Code Review / Codex review 等)

CodeQL によるセキュリティスキャン追加

Sentry 等のエラー追跡導入(本番運用が始まる時)

## 9.4 フェーズ3: 採用と専任化

基盤と運用フローが固まってから、メンバー採用に進む。

採用前に役割定義書を確定する(各役割の責任範囲、評価指標、報告先)

AGENTS.md / CLAUDE.md / 運用手順をオンボーディング資料として整備する

採用順序: 技術メンター(副業) → TL → QA / AI実装・保守 → UI/UXデザイン

採用時は基盤(GitHub組織、テンプレート、ドキュメント)を確認できる状態で迎える

## 9.5 フェーズ4: 拡張運用

チームが安定したら、運用の改善と拡張に進む。

ハーネスの継続改善(TL主導)

ナレッジ整備(QA主導、Notion 等)

AI実装・保守の増員(並列度を上げる)

インシデント対応プロセスの訓練

レトロスペクティブの定例化

## 9.6 採用順序の指針

| 順番 | 役割 | 採用判断の目安 |
| --- | --- | --- |
| 1 | 技術メンター(副業) | 基盤整備の初期段階から相談相手として確保 |
| 2 | TL(技術責任者) | プロダクトが複数になる or 技術判断量が増える時 |
| 3 | QA / AI実装・保守 | 並列AI数が増え、品質判断が追いつかない時 |
| 4 | UI/UXデザイン | ユーザー向けプロダクトを本格化させる時 |
| 5 | AI実装・保守増員 | AI並列度を上げてスケールしたい時 |

| ✅ 最初の一歩<br>完璧を目指さない。フェーズ0で全部やる必要はない。<br>テンプレートリポジトリを1つ作るだけで、以降のプロジェクト立ち上げが大幅に楽になる。<br>AGENTS.md の整備は最初の30日で必ず着手する(以降の全ての投資効果が変わる)。 |
