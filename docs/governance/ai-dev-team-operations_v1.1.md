# 1. 全員が守る基本原則

## 1.1 7つの原則

### 原則1: 真実の源はGitHub

コード、Issue、PR、議論はすべてGitHubに集約する

Slackで議論した重要な決定は必ずGitHub Issueに記録する

「Slackで聞いたはず」「口頭で決めた」は無効。書かれていないことは存在しない

### 原則2: 判断は人間、実装はAI

コードを書く作業はAIに委譲する

人間は「何を作るか」「動いていることを確認」「リリースしてよいか」を判断する

人間がコードを直接タイプするのは緊急時のみ

### 原則3: main は常にデプロイ可能

main ブランチには壊れたコードを入れない

PR は CI 通過を必須とする

壊れたら最優先で修正する(他の作業を止めてでも)

### 原則4: テストは先に書く

仕様 → テスト → 実装の順で進める

AIに「テストも一緒に書いて」を毎回指示する

AGENTS.md に「実装時にテストも併設」を明記する

### 原則5: 1 PR = 1 変更目的

複数の目的を1つのPRに詰め込まない

差分は 200〜400 行が目安

巨大PRはレビューされないと心得る

### 原則6: AI生成は透明化

PR説明欄に「AI生成範囲」を必ず明記する

AIが提案した依存ライブラリは追加前にレビューする

AIにそのまま任せるなと心得る

### 原則7: セキュリティは全工程に統合

シークレットを git に含めない(.gitignore 徹底)

認証情報は 1Password に集約する

不審なら一旦止めて TL / QA に相談する

## 1.2 役割の境界

各役割の責任範囲を明確にする。境界を越える判断が必要なときは、該当役割に確認する。

| 判断の種類 | 責任者 | 相談先 |
| --- | --- | --- |
| 仕様化・アーキ判断 | TL | 発注元 |
| 技術選定 | TL | QA(運用観点) |
| AI実装指示 | TL / AI実装・保守 | — |
| 品質ゲート判断 | QA | TL |
| リリース判断 | QA + 発注元 | TL |
| インシデント対応 | QA(初動) / TL(技術) | 発注元(影響大の場合) |
| セキュリティ判断 | TL + QA | 発注元(影響大の場合) |
| デザイン判断 | UI/UX | TL(実装可否) |

## 1.3 コミュニケーションルール

### 使い分け

| 手段 | 用途 | 応答目安 |
| --- | --- | --- |
| GitHub Issue | 課題、要望、議論 | 1営業日以内 |
| GitHub PR | コードレビュー、変更議論 | 1営業日以内 |
| Slack #dev | 日常の相談、軽い質問 | 数時間以内 |
| Slack #incident | 障害・緊急対応 | 即時 |
| Slack DM | 個人的な相談、機微情報 | 適宜 |
| 対面 / Web会議 | 認識合わせ、意思決定 | 週1の定例 + 必要時 |

### 質問するときの作法

「動かない」だけでは不足。「何をした」「何を期待した」「実際の結果」「試したこと」を書く

エラーメッセージはスクリーンショットではなくテキストでコピペする

コードは Slack の code block(```) で囲む

質問する前に AGENTS.md / RUNBOOK.md を確認する


# 2. 日常開発フロー

## 2.1 全体の流れ

以下のフローを基本とする。GitHub Flow をベースに、AI実装を組み込んだ形である。

[1] 発注元から要件受領(Issue化)
       ↓
[2] TL: 仕様化(設計書 + 受入基準)
       ↓
[3] QA + AI実装・保守: テスト設計
       ↓
[4] AI実装・保守 or TL: AIに実装依頼
       ↓
[5] AI: ブランチ作成・実装・テスト・PR作成
       ↓
[6] CI: 自動検証(Lint・Test・Security)
       ↓
[7] AI レビュー(Claude Code Review / Codex review 等)
       ↓
[8] AI実装・保守: 最終確認
       ↓
[9] マージ(Squash and Merge)
       ↓
[10] 自動デプロイ(ステージング)
       ↓
[11] QA: 品質ゲート
       ↓
[12] 発注元: リリース承認 → 本番昇格

## 2.2 ブランチ運用

### ブランチ命名

| プレフィックス | 用途 | 例 |
| --- | --- | --- |
| feat/ | 新機能追加 | feat/oauth-login |
| fix/ | バグ修正 | fix/login-error-handling |
| refactor/ | リファクタリング | refactor/extract-auth-service |
| docs/ | ドキュメント変更 | docs/update-readme |
| test/ | テスト追加・修正 | test/add-login-edge-cases |
| chore/ | ビルド・補助変更 | chore/upgrade-deps |
| claude/ | Claude Code 自動作成 | claude/issue-42-add-search |
| codex/ | Codex 自動作成 | codex/issue-43 |

### ブランチ運用のルール

main から派生させる(他のブランチからは派生しない)

作業期間は短く保つ(原則1〜3日以内にマージ)

マージ後のブランチは自動削除に任せる

長期化しそうな場合は TL に相談し、小さなPRに分割する

### やってはいけないこと

| ⚠️ ブランチ禁止事項<br>main への直接 push(ブランチ保護で物理的に禁止)<br>他人のブランチへの勝手な push<br>1週間以上放置されたブランチを残す(削除するか、Draft PRを作成)<br>ブランチ名に意味のない文字列(work、tmp、a 等) |
| --- |

## 2.3 コミット運用

### Conventional Commits 規約

コミットメッセージは以下の形式に統一する。

<type>(<scope>): <subject>

<body>

<footer>

type には以下のいずれかを使う。

| type | 意味 | SemVer影響 |
| --- | --- | --- |
| feat | 新機能追加 | MINOR |
| fix | バグ修正 | PATCH |
| docs | ドキュメント変更 | なし |
| style | 整形のみ(動作変更なし) | なし |
| refactor | リファクタリング | なし |
| test | テスト追加・修正 | なし |
| chore | ビルド・補助変更 | なし |
| perf | パフォーマンス改善 | PATCH |
| ci | CI設定変更 | なし |
| security | セキュリティ関連修正 | PATCH |

### コミット例

feat(auth): OAuthログインを追加

Google OAuth 2.0 によるログインを実装。
既存のメール認証と並行運用可能。

Closes #42

fix(api): ページネーション境界値のバグ修正

最終ページ表示時に空配列を返していた問題を修正。
リグレッションテストも追加。

Refs #67

### コミットのルール

1コミット = 1論理的変更

subject は50文字以内、命令形(動詞の原形)

body は何故・何をを書く(どうやってはコードから読める)

関連 Issue は footer に Closes / Refs / Fixes で紐づける

AIが生成したコミットも本ルールに従う(AGENTS.mdで指示)

## 2.4 PR運用

### PR作成

作業が完了したら(または途中でも)PRを作成する

途中でも作成する場合は Draft PR を使う

PRテンプレートの全項目を埋める

関連 Issue を必ずリンクする

レビュアーは CODEOWNERS により自動指定される

### PRテンプレート(必須項目)

## 変更内容
- 何を変えたか(箇条書きで簡潔に)

## 背景・理由
- なぜこの変更が必要か
- 関連 Issue: #XX

## 動作確認
- どうやって動作を確認したか
- 確認した環境

## レビュー観点
- 特に見てほしい点

## AI生成範囲
- [ ] 全てAI生成
- [ ] 一部AI生成(具体的に: ___)
- [ ] 人間が全て作成

## チェックリスト
- [ ] テストを追加した
- [ ] CIが通過している
- [ ] ドキュメントを更新した
- [ ] 影響範囲を確認した

### レビュー運用

| レビュアー | 見る観点 | 応答目安 |
| --- | --- | --- |
| AI(Claude Code Review / Codex review 等) | コード規約、明らかなバグ、典型的な脆弱性 | PR作成後すぐ(自動) |
| AI実装・保守 | テスト不足、観点漏れ、運用影響 | 1営業日以内 |
| TL(必要時) | アーキ判断、技術選定の妥当性 | 1営業日以内 |
| QA(必要時) | 品質ゲート観点、運用観点 | リリース前 |

### PRのライフサイクル

Draft PR 作成(任意)

CI 通過

AI レビュー実施

AI実装・保守による確認(必要に応じて TL/QA)

承認(Approve)

Squash and Merge

ブランチ自動削除

### マージのルール

Squash and Merge を基本とする(1 PR = 1 コミット in main)

Merge commit / Rebase は原則使わない

「自動マージ」設定は使ってよい(CI+承認が揃ったら自動でマージ)

マージ実行は PR 作成者が行う(承認とマージは別操作)

## 2.5 デプロイ運用

### デプロイの流れ

| 環境 | デプロイトリガー | 承認 |
| --- | --- | --- |
| プレビュー | PR作成時(自動) | 不要 |
| ステージング | main マージ時(自動) | 不要 |
| 本番 | リリースタグ作成時 | 発注元 + QA |

### リリースのルール

リリースは原則として平日午前中に実施する(障害対応の余裕確保)

金曜午後・連休前のリリースは禁止(緊急修正以外)

リリース前にチェックリスト(7章参照)を完了させる

リリース後30分は誰かが監視できる体制にする


# 3. AI実装・保守の運用

## 3.1 AI実装・保守の役割

AI実装・保守は、AI実装エージェントとCIの間に立ち、品質と運用を担保する役割である。コードは書かない。AIへの指示と監視に専念する。

| 💡 AI実装・保守がやること / やらないこと<br>やること: アラート対応、AIへの修正指示、テスト不足の発見、エスカレーション判断<br>やらないこと: 自分でコードを書く、設計判断、技術選定<br>迷ったら: TL に相談する |
| --- |

## 3.2 日常業務サイクル

### 1日の流れ(例)

| 時間帯 | 業務 | 確認するもの |
| --- | --- | --- |
| 朝(始業時) | 夜間に発生したアラート確認 | Sentry、PagerDuty、CI失敗通知 |
| 午前 | オープンPRの確認、AI出力の検証 | GitHub PR、CI結果 |
| 午後 | テストカバレッジ確認、AIへの追加指示 | Codecov、CI レポート |
| 夕方(終業前) | 翌日朝への引き継ぎ整理 | 未対応 Issue、進行中 PR |

### 作業の3区分

【監視業務】アラート確認、CI失敗対応、本番障害の初動

【品質業務】PRレビュー、テスト不足の発見、観点リスト整備

【AI指示業務】AIへの修正依頼、追加テスト依頼、リファクタ依頼

## 3.3 AIへの指示パターン

### 基本テンプレート

# AI実装依頼テンプレート

## 何を実装するか
- 機能・修正の概要

## 期待される動作
- 入力 → 出力
- エッジケース

## 制約
- 使ってよい技術
- 使ってはいけない技術
- 既存コードへの影響範囲

## テスト要件
- カバレッジ目標
- 必須テスト観点

## 完了条件
- [ ] テスト全通過
- [ ] CIパス
- [ ] AGENTS.md規約準拠

### CI失敗時の対応指示

# CI失敗時のAI指示例

このPR #42 のCIが失敗しています。
ログ: [URLまたは抜粋を貼る]

## やってほしいこと
1. エラー原因を特定してください
2. 修正してください
3. 修正コミットには「fix: <原因>」のメッセージを付けてください
4. 修正後、追加のリグレッションテストも書いてください

## 制約
- 既存テストの期待値を勝手に変えないでください
- テストを通すために仕様を変えないでください

### テスト追加指示

# テスト追加指示例

src/auth/login.py のカバレッジが65%です。
80%まで上げてください。

## 観点
- 異常系: 不正な入力、null、空文字
- 境界値: 最大長、最小長
- セキュリティ: SQLインジェクション、ブルートフォース
- 同時実行: 複数セッションの整合性

## 制約
- パラメタライズドテストを使う
- モックは外部API依存箇所のみ
- テスト1つあたり1観点に絞る

### リファクタリング指示

# リファクタリング指示例

src/services/order_service.py を読み、以下の問題を修正してください。
- 関数が長すぎる(100行超)
- 責務が混在(注文処理 + 在庫更新 + 通知)
- テスタビリティが低い

## やってほしいこと
1. 責務ごとに関数を分割
2. 既存のテストは全てパスすることを確認
3. 新しい関数にもテストを追加

## やってはいけないこと
- 外部APIの呼び出し方を変える
- DB スキーマを変える
- 既存の公開インタフェースを変える

## 3.4 アラート対応

### アラート種別と対応

| アラート種別 | 発生源 | 初動 | エスカレーション |
| --- | --- | --- | --- |
| CI失敗 | GitHub Actions | PR作成者に通知、AIに修正指示 | 30分で解決しなければ TL |
| 本番エラー | Sentry | 影響範囲確認、緊急度判定 | 影響大なら QA + TL |
| 稼働監視異常 | Uptime Robot | サービス到達性確認 | 5分で解決しなければ TL |
| セキュリティアラート | Dependabot 等 | 影響評価、修正PR起案 | Critical は即 TL |
| パフォーマンス劣化 | Datadog 等 | 原因切り分け、AIに調査指示 | 顧客影響が出る前に TL |

### 初動の判断フロー

アラート受信
   ↓
影響範囲確認(本番影響あり?)
   ├─ Yes → 緊急対応(#incident チャンネルに投稿)
   │         ↓
   │       QA に即時連絡
   │         ↓
   │       影響大なら TL / 発注元 にエスカレーション
   │
   └─ No → 通常対応
            ↓
          AI に原因調査指示
            ↓
          修正PR作成 → レビュー → マージ

## 3.5 観点リストの育て方

### 観点リストとは

AI実装の品質を担保するため、過去の失敗パターンや業務領域固有の観点を蓄積したリスト。これがAI実装・保守の最大の武器となる。

### 観点リストの構造

docs/perspectives/
├── general.md          # 全プロジェクト共通
├── auth.md             # 認証関連の観点
├── payment.md          # 決済関連の観点
├── data-mutation.md    # データ書き込み観点
└── ui.md               # UI関連の観点

### 観点を追加するタイミング

本番障害が発生した時(必ず追加、リグレッションテスト化)

PRレビューで指摘した観点(「自分の指摘」の形式知化)

過去PRで見つけたバグから一般化できた時

新しい技術領域に触れた時

### 観点リストの記述例

# 認証関連の観点

## 正常系
- 正しい認証情報でログイン成功
- セッションが期待通り維持される

## 異常系
- 間違った認証情報でログイン失敗
- 存在しないユーザーでログイン失敗
- ロックされたアカウントでログイン失敗

## 境界値
- パスワード最小長(8文字)
- パスワード最大長(128文字)
- メールアドレス最大長
- 空文字、null、特殊文字

## セキュリティ
- SQLインジェクション(' OR '1'='1 等)
- タイミング攻撃耐性(失敗時の応答時間一定)
- ブルートフォース対策(レート制限、アカウントロック)
- セッション固定化攻撃
- CSRF対策

## 同時実行
- 同一ユーザーの複数セッション
- ログアウト中の認証試行

## 環境依存
- DB接続切断時の挙動
- 認証サーバーダウン時の挙動

## リグレッション
- #234: 半角スペース付きメールが弾かれていた → 修正
- #267: 大文字のメールが区別されていた → 正規化対応


# 4. テスト・品質ゲート

## 4.1 テスト戦略の基本

### テストピラミッド

以下の比率を基本目標とする。

| レベル | 目標比率 | 対象 | 実行時間 |
| --- | --- | --- | --- |
| 静的解析 | 全コード | Lint、型、セキュリティ | 数秒 |
| ユニットテスト | 60-70% | 純粋ロジック、計算、変換 | 数秒〜数十秒 |
| 結合テスト | 20-30% | DB操作、外部API、複数モジュール連携 | 数十秒〜数分 |
| E2Eテスト | 5-10% | 主要ユーザーシナリオ | 数分 |

### テストの優先順位

クリティカルパスのテスト(認証、決済、データ書き込み等)

リグレッションテスト(過去バグの再発防止)

異常系テスト

境界値テスト

正常系テスト

性能テスト

## 4.2 必須テスト観点

すべてのプロダクトコードに対して、以下の観点を網羅する。

| 観点 | 確認内容 | 適用範囲 |
| --- | --- | --- |
| 正常系 | 想定通りの入出力で動作する | 全機能 |
| 境界値 | 空、null、最大、最小、0、-1 | 全機能 |
| 異常系 | 不正入力、型違い、欠損 | 全機能 |
| エラーハンドリング | 例外発生時に適切に処理 | 外部依存あり |
| セキュリティ | 認証・認可・インジェクション | 認証/権限を扱う場合 |
| 並行性 | 同時実行の整合性 | 状態を持つ場合 |
| 性能 | 想定負荷での応答時間 | クリティカルパス |
| 状態遷移 | 状態の正しい移行 | ステートマシン |

## 4.3 カバレッジ基準

### プロダクト分類別の目標

| 分類 | カバレッジ目標 | 備考 |
| --- | --- | --- |
| クリティカルパス | 80% 以上 | 認証、決済、データ書き込み |
| 一般機能 | 50〜70% | 主要なビジネスロジック |
| UI/プレゼンテーション | E2Eで代替 | ピクセル単位の試験は不要 |
| 設定・ボイラープレート | 対象外 | テストする意味が薄い領域 |

### カバレッジ運用のルール

カバレッジは「指標」であって「目標」ではない

意味のないテストでカバレッジを上げない(関数の存在確認等)

テスト1つにつき1観点を確認する

パラメタライズドテストを優先する(同じ観点を複数値で試す)

カバレッジが下がるPRはCIで弾かれる

## 4.5 テスト失敗時の対応

### 基本フロー

失敗テストのログを確認

ローカル or CIで再現確認

原因を特定(自分の変更 / 他要因 / Flaky)

対応方針を決定(後述)

修正PR作成または相談

### 原因別の対応

| 原因 | 対応 | 禁止事項 |
| --- | --- | --- |
| 自分の変更が原因 | AI に修正依頼 → 再CI | テストの期待値を変える、テストをスキップする |
| 既存バグの顕在化 | 別Issueとして起票し、別PRで修正 | 今回のPRに含める |
| Flaky test | 原因調査(タイミング、外部依存)、再現確認 | 再実行で逃げる、無視する |
| 環境問題 | TL に相談、再現条件を記録 | 「自分の環境では動く」で済ます |
| 仕様変更 | テストを正しく更新 | テストを削除する |

### やってはいけないこと

| ⚠️ テスト関連禁止事項<br>失敗するテストを削除する(原因を見ずに)<br>skipアノテーションで一時的に逃げる(理由なくは禁止)<br>カバレッジ閾値を下げる(全員合意なくは禁止)<br>「ローカルで動く」を理由にCIエラーを放置する<br>AI生成のテストを内容確認せずマージする |
| --- |

## 4.6 リグレッションテストの蓄積

### 基本ルール

本番障害が発生したら、必ず再現テストを追加する

PRで指摘されたバグも同様に再現テストを書く

「テスト追加 = 知識の形式知化」と捉える

### リグレッションテスト記述例

```python
def test_regression_234_email_with_leading_space():
    """
    Regression for #234.
    
    Background:
        Emails with leading/trailing whitespace were rejected,
        even though they were valid user inputs.
    
    Expected:
        Trim the whitespace and treat the email as valid.
    """
    response = login(" test@example.com ", "password")
    assert response.success == True
```

### リグレッションテストの管理

テスト関数名に Issue/PR 番号を含める(後で追跡可能に)

docstring に背景・期待動作を必ず記載

削除する場合は理由を PR 説明に明記する

## 5.2 シークレット漏洩発生時の対応

### 発見〜初動(発見後 30分以内)

発見者は即座に Slack #incident に投稿

漏洩したシークレットを無効化(ローテーション)

該当コミットを git history から削除(後述)

影響範囲を特定(誰がアクセス可能だったか)

### シークレットを git から削除する手順

```bash
# 1. 該当ファイルを履歴から削除
git filter-repo --path <ファイルパス> --invert-paths

# 2. リモートに強制push(チームメンバー全員に事前通知)
git push origin --force --all

# 3. シークレットを無効化(必須・最優先)
#   - GitHub Settings から該当 Secret を削除
#   - 該当サービス側で API キーを無効化
```
#   - 新しい認証情報を発行

# 4. インシデント報告書を作成

| ⚠️ 重要<br>git からの削除「だけ」では不十分。漏洩したシークレットは必ず無効化する。<br>git history を書き換える前に、チーム全員に通知する。<br>公開リポジトリでの漏洩は外部からも見られている前提で対応する。 |

## 6.2 緊急パッチの手順

### ホットフィックスの流れ

本番障害対応など、通常のレビューサイクルを待てない場合の手順。

#incident で緊急パッチである旨を宣言

hotfix/<issue> ブランチを main から作成

最小限の修正を実装

PR作成(緊急パッチであることを説明欄に明記)

CIは通す(これだけは省略しない)

レビュアー1名以上の即時承認(通常時より少なくてOK)

マージ・本番デプロイ

収束後、通常のレビュー観点で事後レビュー実施

### ホットフィックスのルール

| ⚠️ 緊急時こそ最小限の変更<br>修正は最小限に絞る(関連する改善は別PRで)。<br>CIは絶対に通す。Lintやテストをスキップしない。<br>事後レビューを必ず実施し、観点リストに追加する。<br>緊急パッチが頻発するなら、根本的な問題がある。 |
| --- |

## 6.3 ロールバック手順

### デプロイ起因の障害

#incident で「ロールバック実施」を宣言

前バージョンのリリースタグを確認

該当環境にロールバック実行(自動化されているはず)

動作確認(主要シナリオ)

収束を宣言

原因調査・修正(通常フローで実施)

### DB変更を含む場合

| ⚠️ DB変更は単純なロールバックが効かない<br>マイグレーションは「前進」「後退」両方のスクリプトを用意する。<br>破壊的変更(列削除等)は段階的に行う(削除はリリース数回後)。<br>ロールバック判断は QA + TL で実施する。<br>データ復旧が必要な場合は、バックアップ手順を確認する。 |
| --- |

## 6.4 連絡が取れない時

### キーパーソン不在時の判断

TL不在: QA が技術判断も代行(慎重に)

QA不在: AI実装・保守 が初動、リリースは保留

発注元不在: 顧客影響を判断、不可逆な変更は保留

全員不在(深夜等): 影響軽微なら翌朝、重大なら誰かに連絡

### 連絡手段の階段

Slack(通常)

Slack DM(直接)

メール

電話(緊急時のみ)
